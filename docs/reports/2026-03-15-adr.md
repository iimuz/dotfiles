# AIエージェント駆動開発におけるドキュメント・ADR管理戦略

## 1. 基本哲学：真実の源（Source of Truth）の再定義

AIエージェントを用いた開発において、人間が手書きでメンテナンスする「状態（What/How）を説明するドキュメント」は必ず腐敗し、AIに嘘の文脈（ハルシネーション）を与える猛毒となる。
したがって、リポジトリに残すべき真実は以下の2つのみに絞り込まれる。

1. **実行可能なアーティファクト**: ソースコード、型定義、テストコード、リンター設定（現在のWhat/How）
2. **不変の決定履歴（ADR）**: なぜその設計にしたのかという理由や過去の痛み（不変のWhy）

## 2. ディレクトリ・ファイル別の正しい扱い方

| ディレクトリ | 扱い方                            | 理由・運用ルール                                                                                                                                                     |
| :----------- | :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `adr/`       | **永久に残す**                    | 過去の決定と理由（Why）。AIの先祖返りや過剰最適化を防ぐ命綱。                                                                                                        |
| `plans/`     | **Doneで削除**                    | 実装完了後、状態はコードに吸収されるため不要。過程で得た重要なWhyのみADRに抽出して捨てる。                                                                           |
| `design/`    | **手書きは削除 / 自動生成は許容** | 人間が書く設計書は腐敗するため実装後に捨てる。Deep Wikiのように「コードからAI/CIが毎回自動生成するRead-Onlyの地図」であれば許容される（※人間が加筆してはいけない）。 |
| `debt/`      | **コード内へ移行**                | 負債リストはコードと乖離するため、該当箇所のソースコードに `// TODO(debt):` として直接埋め込む。                                                                     |
| `reports/`   | **AIから隠蔽**                    | カバレッジ等の記録。人間用としては残してよいが、AIのノイズになるため `.claudesignore` 等で検索対象外にする。                                                         |

## 3. ADRに残す「粒度」と判断基準（3つのテスト）

ADRに残すかどうかの基準は「対象の大きさ（アーキテクチャかCIの構造か）」ではなく、**「AIがその『Why』を知らずに変更を加えた際、致命的な事故が起きるか」**で判断する。

1. **「AIの無邪気な最適化」テスト**: AIに「綺麗にして」と指示したとき、良かれと思って構造を壊し、バグやコスト増を招くか？（YesならADRへ）
2. **「血の教訓（ペイン）」テスト**: その決定は、過去の障害や開発体験の悪化といった「痛み」を伴って学んだ回避策か？（YesならADRへ）
3. **「代替表現の可否」テスト**: 該当コードのすぐ上に `// WHY:` コメントを書くだけで十分伝わるか？（YesならADRは作らずインラインコメントで済ませる）

## 4. 過去に破棄されたADRを残す理由

「現在の決定」だけでなく、**「過去に採用し、現在は破棄された決定（Deprecated / Superseded）」**も書き換えずに残す必要がある。

- **AIの「先祖返り」防止**: 「過去にそれを試したが失敗した」という歴史（負のコンテキスト）を知らないと、AIが再び過去の失敗案を「改善案」として提案してくるため。
- **変遷の理解**: 「なぜ今の複雑なルールがあるのか」という文脈をAIに理解させるため。
- **構造化による腐敗防止**: ADRはステータス（Accepted / Superseded等）を持つため、過去の記録として明示されていれば、AIが現在の仕様と勘違いするリスクはない。

## 5. 2種類のADR：技術的ADRとビジネスADR（BDR）

AIエージェントの暴走を止めるため、ADRは守る対象によって2種類に分かれる。これらを1ファイル1ADRのMarkdownで管理し、AI向けに**「🤖 AI Guardrails」**セクションを設けるのがベストプラクティス。

### A. 通常のADR（技術的ADR）

- **目的**: AIが「一般的なベストプラクティス」を盲信し、現在のシステムに合わない技術的変更を行うのを防ぐ。（例：CIの直列化、ライブラリ選定）
- **フォーマット例**:

  ```markdown
  # ADR-0015: 状態管理にZustandを採用し、Reduxを廃止する

  **Status:** Accepted
  **Date:** 2026-03-14
  **Tags:** [Architecture, Frontend]

  ## Context / Decision

  Reduxのボイラープレートの多さとバンドルサイズ肥大化のペインを解消するため、Zustandを採用する。

  ## 🤖 AI Guardrails (AIエージェントへの指示・禁止事項)

  - **禁止事項**: 既存のZustandストアをReduxやContext APIに書き換える提案・実行をしてはならない。
  - **理由**: バンドルサイズの削減という過去の意思決定を保護するため。
  ```

### B. ビジネスADR（BDR: Business Decision Record）

- **目的**: AIが「コードの短縮・最適化」を優先し、現実世界のルール（法律、コンプライアンス、特殊な顧客要件）を誤って削除してしまうのを防ぐ。
- **フォーマット例**:

  ```markdown
  # BDR-0004: 酒類購入時の年齢確認と決済ブロックの厳格化

  **Status:** Accepted
  **Date:** 2026-03-14
  **Domain:** [Checkout, Compliance]

  ## Business Rule / Decision

  未成年者飲酒禁止法に基づき、酒類カート決済時は年齢確認フラグのみを評価し、親権者同意等の他の救済フラグを完全に無視してブロックするハードコーディングを実装する。

  ## 🤖 AI Guardrails (AIエージェントへの指示・禁止事項)

  - **絶対禁止事項**: この「酒類に対する例外的な処理」を、冗長であるという理由で削除したり、他の処理に共通化してはならない。
  - **優先順位**: コードの美しさよりも、この法的要件の厳守を最優先（Highest Priority）とすること。
  - **理由**: これをバイパスすると法律違反となり、サービス停止等の致命的リスクに直結するため。
  ```

## 6. このアプローチに対する賛否とハイブリッドな着地点

この「コードとADRのみを残す」戦略には、現在のAIエージェント界隈でも議論がある。

- **🟢 賛成派（推進派）の意見**:
  - 古いDesignドキュメントによるAIのハルシネーションを物理的に防げる。
  - コンテキストウィンドウ（トークン）を節約し、推論精度が向上する。
  - AIはコードを逆アセンブルして構造を理解できるため、人間用の要約は不要。
- **🔴 反対派（慎重派）の意見**:
  - 人間のエンジニアの認知負荷が高まり、オンボーディングが困難になる。
  - ビジネス上のドメイン知識や、マクロなアーキテクチャの文脈が失われやすい。

**【現在のベストな着地点】**
「真実の源はコードとADR」という原則を維持しつつ、人間のための俯瞰図は「具体的な仕様を書かない概念モデル」のみに留めるか、Deep Wikiのように「CI/AIがコードから100%自動生成するRead-Onlyの地図」として運用することで、AIの精度と人間の理解を両立させる。
