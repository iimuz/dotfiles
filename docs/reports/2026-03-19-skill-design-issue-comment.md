# Skill Design Report: issue-comment

## 目的

GitHub Issue にサマリ + 折りたたみ詳細レポートの形式でコメントを投稿する Skill を作成する。

## 実施した内容

1. 調査結果を Markdown レポートとしてセッション内で作成
2. レポートからサマリ（要約テーブル + 修正方針 1-2 行）を抽出
3. `gh issue comment` コマンドで以下の構造のコメントを投稿:
   - サマリ（常に表示）
   - `<details><summary>詳細レポート</summary>` で囲んだ本文（折りたたみ）

## 意図

- Issue のコメント欄が長文で埋まると可読性が下がる
- サマリだけ読めば概要を把握でき、詳細が必要な場合のみ展開する
- 調査・分析結果を Issue に記録として残す際の標準フォーマットを統一する

## Skill に含めるべきプロンプト要素

### 入力パラメータ

- `owner`: リポジトリオーナー
- `repo`: リポジトリ名
- `issue_number`: Issue 番号
- `summary`: サマリ（Markdown。見出し + 短い説明 + テーブルなど。常に表示される部分）
- `details`: 詳細レポート（Markdown。折りたたみ内に入る本文）
- `details_label` (optional): `<summary>` タグのラベル。デフォルト: `詳細レポート`

### 出力フォーマット

```markdown
{summary}

<details>
<summary>{details_label}</summary>

{details}

</details>
```

### プロンプトに含めるべき指示

1. **構造規則**:
   - summary は常に表示されるため、簡潔にまとめる（目安: 10 行以内、テーブル 1 つまで）
   - details は折りたたみ内に入るため、長文・テーブル・コードブロックを自由に使って良い
   - `<details>` タグの前後に空行を入れる（GitHub Markdown のレンダリング要件）
   - `<summary>` タグの後に空行を入れる（同上）
   - `</details>` の前に空行を入れる（同上）

2. **内容規則**:
   - summary には結論・方針・件数サマリを含める
   - details には根拠・分析過程・既存状況・追加すべき項目の詳細を含める
   - summary だけ読んでも次のアクションが分かるようにする

3. **投稿規則**:
   - `gh issue comment {issue_number} --body '{content}'` で投稿する
   - body 内にシングルクォートが含まれる場合は適切にエスケープする
   - 投稿前に dry-run として内容を確認する（実際の Issue には書き込まない）
   - ユーザーの明示的な確認後に投稿する

4. **言語規則**:
   - コメントは日本語で記述する（プロジェクトの規約に従う）

### コマンド実行例

```bash
gh issue comment 2448 --body '## サマリ見出し

要約テキスト...

| 列1 | 列2 |
|---|---|
| 値1 | 値2 |

<details>
<summary>詳細レポート</summary>

## 詳細セクション 1

本文...

## 詳細セクション 2

本文...

</details>'
```

### エッジケース対応

- body にシングルクォート `'` が含まれる場合: ヒアドキュメント (`<<'EOF'`) を使用する
- body が長すぎる場合 (>65536 文字): 分割投稿するか、要約を強化して details を短縮する
- Markdown テーブル内の `|`: エスケープ不要（GitHub がテーブル内と認識する）
- details 内の HTML タグ: GitHub Markdown で許可されたタグのみ使用する

## Skill の呼び出しタイミング

- 調査・分析結果を Issue に記録する際
- PR レビュー結果を Issue にまとめる際
- 障害調査の報告を Issue に投稿する際
- いずれも「サマリ + 詳細」の 2 層構造が適切な場合に使用する
